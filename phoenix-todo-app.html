Prerequisites & Setup

Before we start, make sure you have these installed:

  • Elixir 1.14+ (Installation guide)
  • Phoenix 1.7+ 
    mix archive.install hex phx_new
  • PostgreSQL running locally (or use SQLite for simplicity)
  • Node.js for asset compilation
Troubleshooting tip: If you get "command not found" errors, make sure Elixir's bin directory is in your PATH. Run elixir --version to verify your installation.

Part 1: Project Setup & Initial Configuration

Step 1: Create Your Phoenix Project

🎯 What we're doing: Creating a new Phoenix project with LiveView enabled and PostgreSQL as the database. 
mix phx.new live_todo --live --database postgres
📋 Command breakdown:
  • mix phx.new - Phoenix generator command that creates a new application
  • live_todo - Your app name (creates a folder with this name and sets the module name)
  • --live - Includes Phoenix LiveView for real-time, interactive features without JavaScript
  • --database postgres - Configures PostgreSQL as the database (alternatives: mysql, sqlite3)
💡 Alternative for beginners: If you want to avoid PostgreSQL setup, use: 
mix phx.new live_todo --live --database sqlite3
✅ Checkpoint: You should see a new live_todo folder with the Phoenix project structure. If you see any errors, double-check your Phoenix installation.

Step 2: Install Dependencies and Setup Database

cd live_todo
mix deps.get          # Downloads all required packages
mix ecto.setup        # Creates database + runs migrations + seeds
📋 What each command does:
  • cd live_todo - Navigate into your new project directory
  • mix deps.get - Downloads and installs all dependencies listed in mix.exs (Ecto for database, Phoenix LiveView, etc.)
  • mix ecto.setup - Runs three commands in sequence:
    • mix ecto.create - Creates the database
    • mix ecto.migrate - Runs any existing migrations
    • mix run priv/repo/seeds.exs - Seeds the database with initial data
🔍 What's happening behind the scenes:

Phoenix is setting up your development environment with all the tools you need: database connection, LiveView for real-time features, and a complete web framework structure.

Database connection issues?
  • Make sure PostgreSQL is running: brew services start postgresql (macOS)
  • Check your database config in config/dev.exs
  • For SQLite, use --database sqlite3 when creating the project

Step 3: Test Your Setup

mix phx.server

Visit http://localhost:4000 - you should see the Phoenix welcome page!

🎉 Great job! You have a working Phoenix application. Press Ctrl+C twice to stop the server when you're ready to continue.

Part 2: Adding User Authentication

Why authentication first? We're adding user authentication before building our todo features because each user should only see their own todos. Phoenix makes this incredibly easy with built-in generators.

Step 4: Generate Authentication System

mix phx.gen.auth Accounts User users
📋 Command breakdown:
  • mix phx.gen.auth - Phoenix's built-in authentication generator
  • Accounts - The context name (business logic module)
  • User - The schema name (data structure)
  • users - The database table name
🏗️ This command creates:
  • Accounts context - Functions for managing users (lib/live_todo/accounts.ex)
  • User schema - Database structure for users (lib/live_todo/accounts/user.ex)
  • Authentication pages - Login, register, password reset (LiveView pages)
  • Database migrations - Tables for users and sessions
  • Security plugs - Middleware for protecting routes
  • Tests - Complete test suite for authentication
🔐 Security features included:
  • Password hashing with Bcrypt
  • Email confirmation
  • Password reset functionality
  • Session management
  • Remember me functionality
Run the new migrations: 
mix ecto.migrate

Part 3: Creating Todo Lists

Our data model: Each user can have multiple todo lists, and each list can have multiple todo items. This is called a "one-to-many" relationship in database terms.

Step 7: Generate Todo List CRUD

🎯 Create the todo list functionality: 
mix phx.gen.live Todos TodoList todo_lists name:string description:text
📋 Command breakdown:
  • mix phx.gen.live - Generates a complete LiveView CRUD interface
  • Todos - Context name (business logic module)
  • TodoList - Schema name (singular, CamelCase)
  • todo_lists - Table name (plural, snake_case)
  • name:string description:text - Field definitions
🏗️ This creates:
  • Todos context - Business logic for todos (lib/live_todo/todos.ex)
  • TodoList schema - Database structure (lib/live_todo/todos/todo_list.ex)
  • LiveView pages - Index, show, and form components
  • Database migration - Creates todo_lists table
  • Tests - Complete test suite for the functionality
💡 LiveView vs Controllers: We're using phx.gen.live instead of phx.gen.html because LiveView provides real-time updates without page refreshes. Perfect for interactive todo lists!

Step 9: Update Schemas with Associations

Update TodoList schema (lib/live_todo/todos/todo_list.ex): 

defmodule LiveTodo.Todos.TodoList do
  use Ecto.Schema
  import Ecto.Changeset

  schema "todo_lists" do
    field :name, :string
    field :description, :string
    # ADD THIS LINE:
    belongs_to :user, LiveTodo.Accounts.User

    timestamps()
  end

  @doc false
  def changeset(todo_list, attrs) do
    todo_list
    |> cast(attrs, [:name, :description])
    |> validate_required([:name])

    |> validate_length(:name, min: 1, max: 100)
  end
end

Update User schema (lib/live_todo/accounts/user.ex): 

# Add this line inside the schema block:

Step 10: Update Context to Scope by User

Update the Todos context (lib/live_todo/todos.ex) to only show each user's lists: 

defmodule LiveTodo.Todos do
  import Ecto.Query, warn: false
  alias LiveTodo.Repo
  alias LiveTodo.Todos.TodoList
  alias LiveTodo.Accounts.User

  # Update this function:
  def list_todo_lists(%User{id: user_id}) do
    from(l in TodoList, where: l.user_id == ^user_id, order_by: [desc: l.inserted_at])
    |> Repo.all()
  end

  # Update this function:
  def get_todo_list!(%User{id: user_id}, id) do
    Repo.get_by!(TodoList, id: id, user_id: user_id)
  end

  # Update this function:
  def create_todo_list(%User{} = user, attrs \\ %{}) do
    %TodoList{}
    |> TodoList.changeset(attrs)
    |> Ecto.Changeset.put_assoc(:user, user)
    |> Repo.insert()
  end

  # Update this function:
  def update_todo_list(%User{id: user_id}, %TodoList{user_id: user_id} = todo_list, attrs) do
    todo_list
    |> TodoList.changeset(attrs)
    |> Repo.update()
  end

  # Update this function:
  def delete_todo_list(%User{id: user_id}, %TodoList{user_id: user_id} = todo_list) do
    Repo.delete(todo_list)
  end
end
🔐 Security note: Notice how we're checking user_id in every function? This ensures users can only access their own todo lists! This is called "authorization" - making sure users can only access data they own. Never skip this step in real applications.

defmodule LiveTodoWeb.TodoListLive.Index do
  use LiveTodoWeb, :live_view

  alias LiveTodo.Todos
  alias LiveTodo.Todos.TodoList

  @impl true
  def mount(_params, _session, socket) do
    # Get current user from socket assigns
    current_user = socket.assigns.current_user
    todo_lists = Todos.list_todo_lists(current_user)

    {:ok, assign(socket, :todo_lists, todo_lists)}
  end

  # Update other functions to pass current_user:
  @impl true
  def handle_info({LiveTodoWeb.TodoListLive.FormComponent, {:saved, todo_list}}, socket) do
    current_user = socket.assigns.current_user
    {:noreply, assign(socket, :todo_lists, Todos.list_todo_lists(current_user))}
  end

  @impl true
  def handle_event("delete", %{"id" => id}, socket) do
    current_user = socket.assigns.current_user
    todo_list = Todos.get_todo_list!(current_user, id)
    {:ok, _} = Todos.delete_todo_list(current_user, todo_list)

    {:noreply, assign(socket, :todo_lists, Todos.list_todo_lists(current_user))}
  end
end

Part 4: Adding Todo Items

Next level: Now we'll add individual todo items to each list. This creates another one-to-many relationship: each todo list has many todo items.

Step 14: Generate Todo Items

mix phx.gen.live Todos TodoItem todo_items \
  title:string \
  description:text \
  completed:boolean \
  todo_list_id:references:todo_lists
📋 Command breakdown:
  • TodoItem - Individual todo item schema
  • todo_items - Database table name
  • title:string - The todo item title
  • description:text - Optional longer description
  • completed:boolean - Whether the item is done
  • todo_list_id:references:todo_lists - Foreign key to todo_lists table
🔗 Database relationships: The references:todo_lists creates a foreign key relationship. This means each todo item belongs to exactly one todo list.

Step 15: Update TodoItem Schema

    |> validate_required([:title])
    |> validate_length(:title, min: 1, max: 200)
  end
end

Step 18: Update the Show LiveView

Update lib/live_todo_web/live/todo_list_live/show.ex: 

defmodule LiveTodoWeb.TodoListLive.Show do
  use LiveTodoWeb, :live_view

  alias LiveTodo.Todos
  alias LiveTodo.Todos.TodoItem

  @impl true
  def mount(_params, _session, socket) do
    {:ok, socket}
  end

  @impl true
  def handle_params(%{"id" => id}, _, socket) do
    current_user = socket.assigns.current_user
    todo_list = Todos.get_todo_list!(current_user, id)
    todo_items = Todos.list_todo_items(todo_list)

    {:noreply,
     socket
     |> assign(:page_title, page_title(socket.assigns.live_action))
     |> assign(:todo_list, todo_list)
     |> assign(:todo_items, todo_items)
     |> assign(:item_form, to_form(Todos.change_todo_item(%TodoItem{})))}
  end

  @impl true
  def handle_event("save_item", %{"todo_item" => item_params}, socket) do
    case Todos.create_todo_item(socket.assigns.todo_list, item_params) do
      {:ok, _item} ->
        todo_items = Todos.list_todo_items(socket.assigns.todo_list)
        {:noreply,
         socket
         |> assign(:todo_items, todo_items)
         |> assign(:item_form, to_form(Todos.change_todo_item(%TodoItem{})))
         |> put_flash(:info, "Item added successfully!")}

      {:error, %Ecto.Changeset{} = changeset} ->
        {:noreply, assign(socket, :item_form, to_form(changeset))}
    end
  end

  @impl true
  def handle_event("toggle_item", %{"id" => id}, socket) do
    item = Todos.get_todo_item!(id)
    {:ok, _} = Todos.update_todo_item(item, %{completed: !item.completed})

    todo_items = Todos.list_todo_items(socket.assigns.todo_list)
    {:noreply, assign(socket, :todo_items, todo_items)}
  end

  @impl true
  def handle_event("delete_item", %{"id" => id}, socket) do
    item = Todos.get_todo_item!(id)
    {:ok, _} = Todos.delete_todo_item(item)

    todo_items = Todos.list_todo_items(socket.assigns.todo_list)
    {:noreply, assign(socket, :todo_items, todo_items)}
  end

  defp page_title(:show), do: "Show Todo list"
  defp page_title(:edit), do: "Edit Todo list"
end

def list_todo_items(%TodoList{id: todo_list_id}) do
  from(i in TodoItem,
    where: i.todo_list_id == ^todo_list_id,

  todo_item
  |> TodoItem.changeset(attrs)

  Repo.delete(todo_item)
end

Step 23: Writing Your First Test

Let's add a simple test for our todo functionality:

# test/live_todo/todos_test.exs
defmodule LiveTodo.TodosTest do
  use LiveTodo.DataCase
  alias LiveTodo.{Todos, Accounts}

  describe "todo_lists" do
    test "create_todo_list/2 creates a list for a user" do
      user = user_fixture()
      attrs = %{name: "My Todo List", description: "Test description"}

      assert {:ok, todo_list} = Todos.create_todo_list(user, attrs)
      assert todo_list.name == "My Todo List"
      assert todo_list.user_id == user.id
    end

    test "list_todo_lists/1 returns only user's lists" do
      user1 = user_fixture()
      user2 = user_fixture()

      {:ok, list1} = Todos.create_todo_list(user1, %{name: "User 1 List"})
      {:ok, _list2} = Todos.create_todo_list(user2, %{name: "User 2 List"})

      user1_lists = Todos.list_todo_lists(user1)
      assert length(user1_lists) == 1
      assert hd(user1_lists).id == list1.id
    end
  end

  defp user_fixture(attrs \\ %{}) do
    {:ok, user} =
      attrs
      |> Enum.into(%{
        email: "user#{System.unique_integer()}@example.com",
        password: "hello world!"
      })
      |> Accounts.register_user()

    user
  end
end
